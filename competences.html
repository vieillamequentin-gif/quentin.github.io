AC25.01Cyber | Administrer les protections contre les logiciels malveillants

Dans le cadre de la SAÉ 3.Cyber.04 – Découvrir le pentesting, j'ai abordé l'administration des protections contre les logiciels malveillants sous l'angle offensif : comprendre précisément comment un attaquant ou un logiciel malveillant opère est la méthode la plus efficace pour définir, prioriser et mettre en place des protections adaptées.

1. Vérification de la connectivité et de la configuration réseau

Avant toute chose, j'ai vérifié que ma machine Kali Linux était correctement configurée sur le réseau de test avec la commande ip a. Cette commande affiche les interfaces réseau disponibles, leurs adresses IP, masques de sous-réseau, et l'état de chaque interface (UP/DOWN). Ma machine Kali était configurée avec l'adresse 10.15.250.129, dans le même segment réseau que la machine cible (10.16.20.187). Cette étape préliminaire est indispensable : un scan nmap sur une adresse injoignable retourne simplement que l'hôte est "down" sans distinguer si la machine est éteinte ou si le réseau est mal configuré.

2. Identification de la surface d'attaque avec nmap

J'ai réalisé un scan complet de la machine cible (10.16.20.187) avec nmap. nmap (Network Mapper) est l'outil de référence pour la reconnaissance réseau : il envoie des paquets TCP, UDP ou ICMP et analyse les réponses pour déterminer quels ports sont ouverts, quels services y tournent, et quelles versions de ces services sont déployées. Un port ouvert qui répond avec un SYN-ACK à une sonde TCP SYN est un service actif qu'un logiciel malveillant peut exploiter. Ce scan m'a révélé deux points d'entrée potentiels : le port 80 (HTTP) ouvert et accessible, et le port 22 (SSH) dans un état "filtré" (un firewall ou iptables bloque activement ce port, mais le service SSH existe).

Cette démarche reproduit exactement la phase de reconnaissance d'un logiciel malveillant ou d'un worm de type Morris : il scanne le réseau à la recherche de services vulnérables (versions obsolètes de SSH, serveurs HTTP sans authentification, services RPC exposés, etc.) pour identifier ses vecteurs d'infection.

3. Identification des vecteurs d'intrusion exploités

La session complète m'a permis d'identifier quatre vecteurs d'intrusion distincts présents sur cette machine, chacun correspondant à un type de protection spécifique à administrer :

Vecteur 1 – Information Disclosure HTTP : Des credentials (nom d'utilisateur et mot de passe encodé en Base64) étaient accessibles directement dans le code source d'une page web publique. Un logiciel malveillant ou un crawler automatisé peut récupérer ces informations en quelques millisecondes. Protection à administrer : interdire systématiquement l'inclusion de données sensibles dans les pages HTML, utiliser des variables d'environnement ou des gestionnaires de secrets (Vault, AWS Secrets Manager) pour les credentials.

Vecteur 2 – Encodage faible (Base64) : Le mot de passe était protégé par un simple encodage Base64, trivialement réversible avec echo "chaine" | base64 -d. Protection à administrer : stocker les mots de passe avec un algorithme de hachage robuste adapté à cet usage (bcrypt avec un facteur de coût d'au moins 12, ou argon2id), jamais avec un encodage réversible ou un chiffrement symétrique dont la clé serait sur la même machine.

Vecteur 3 – Port SSH non protégé par authentification forte : Malgré le port knocking, une fois la séquence connue, le service SSH acceptait une connexion par mot de passe simple. Protection à administrer : désactiver l'authentification par mot de passe SSH (PasswordAuthentication no dans /etc/ssh/sshd_config), n'autoriser que les clés publiques, et combiner avec fail2ban pour bloquer les tentatives de brute force.

Vecteur 4 – Droits sudo mal configurés (sudo sur vim) : L'utilisateur disposait de droits sudo sans mot de passe sur /usr/bin/vim, un éditeur interactif permettant d'exécuter des commandes shell. Protection à administrer : n'accorder des droits sudo qu'aux commandes strictement nécessaires et sans fonctionnalité de shell intégré, en utilisant des règles /etc/sudoers granulaires avec l'utilitaire visudo. Consulter GTFOBins avant d'accorder des droits sudo sur n'importe quel binaire.

4. Contre-mesures concrètes à administrer

En synthèse, cette expérience m'a permis de dresser une liste concrète de protections à déployer pour réduire la surface d'attaque d'un système Linux : désactiver tous services non nécessaires, ne jamais exposer de credentials dans le code source, utiliser bcrypt/argon2 pour les mots de passe, restreindre sudo aux binaires non interactifs, déployer fail2ban pour SSH, activer les journaux de connexion et mettre en place des alertes (SIEM) sur les connexions SSH réussies depuis des IPs inconnues.

AC25.02Cyber | Prendre en main des outils de test de pénétration réseau/système

Dans le cadre de la SAÉ 3.Cyber.04 – Découvrir le pentesting, j'ai conduit une session de test d'intrusion complète et méthodique sur une machine cible (10.16.20.187) depuis Kali Linux, en enchaînant six étapes couvrant l'intégralité de la kill chain offensive.

Étape 1 – Reconnaissance réseau avec nmap

J'ai exécuté nmap 10.16.20.187 -all pour effectuer un scan complet de la machine cible. nmap détermine l'état de chaque port en analysant les réponses au niveau TCP/IP :

  • Un port ouvert répond avec TCP SYN-ACK à une sonde TCP SYN.
  • Un port fermé répond avec TCP RST (le service n'écoute pas).
  • Un port filtré ne répond pas ou renvoie ICMP unreachable (un firewall bloque activement la sonde, mais quelque chose existe derrière).

Résultat : port 80/tcp (HTTP) ouvert, port 22/tcp (SSH) filtré. Le port 22 filtré indique la présence d'un mécanisme de protection dynamique – il s'agit ici de port knocking implémenté via des règles iptables avec le module recent.

Étape 2 – Extraction d'informations sensibles avec curl

J'ai utilisé curl http://10.16.20.187 pour récupérer le contenu HTML complet de la page web hébergée sur la cible. curl (Client URL) est un outil en ligne de commande permettant de transférer des données via de nombreux protocoles (HTTP, HTTPS, FTP, SFTP…). Il retourne directement le corps de la réponse HTTP dans le terminal, sans interprétation HTML. En scrollant dans la réponse jusqu'au bas de la page, j'ai repéré deux variables contenant ce qui ressemblait à un nom d'utilisateur et un mot de passe encodé. La présence du caractère = ou == en fin de chaîne est la signature caractéristique du padding Base64 : Base64 encode 3 octets en 4 caractères ASCII, et complète avec des = si la longueur des données source n'est pas un multiple de 3. J'ai décodé la chaîne dans le terminal avec plusieurs passes successives : 

echo "chaine_encodee_base64" | base64 -d
# Si le résultat est encore en Base64 :
echo "resultat_intermediaire" | base64 -d
# Répéter jusqu'à obtenir du texte clair

Le résultat final était le mot de passe en clair : zulululu123. Cette étape illustre une vulnérabilité OWASP A02:2021 – Cryptographic Failures : l'utilisation d'un encodage réversible au lieu d'un vrai mécanisme cryptographique pour protéger des données sensibles.

Étape 3 – Résolution du problème de synchronisation SSH (Port Knocking)

Le port knocking est un mécanisme de sécurité par obscurité : le service SSH reste invisible (port filtré) jusqu'à ce qu'une séquence précise de tentatives de connexion TCP sur des ports spécifiques soit détectée par le pare-feu. Une fois la séquence envoyée, le firewall ajoute dynamiquement une règle iptables ouvrant le port 22 pour l'IP source, mais seulement pour une courte durée (typiquement 10-30 secondes). La difficulté : le temps d'exécuter la commande SSH après le port knocking, le port se refermait. La solution technique que j'ai mise en place utilise deux terminaux en parallèle : 

# Terminal 1 (en attente active) :
ssh [email protected]
# Le client SSH essaie de se connecter en boucle

# Terminal 2 (déclenchement du port knocking) :
nmap 10.16.20.187 -r
# L'option -r scanne les ports dans l'ordre, ce qui déclenche la séquence

Au moment précis où nmap a envoyé les paquets de la séquence de port knocking, le pare-feu a ouvert le port 22 pour mon IP. La connexion SSH du terminal 1, qui attendait déjà, a établi la poignée de main TCP en quelques millisecondes avant que le port ne se referme.

Étape 4 – Authentification SSH et exploration du système

La connexion SSH s'est établie avec le compte k1tt3ns et le mot de passe zulululu123 découvert à l'étape 2. Une fois connecté, j'ai exploré le système avec les commandes de base : whoami (confirmer l'identité), id (groupes et UID/GID), pwd (répertoire courant), ls -la (lister les fichiers avec droits et métadonnées), cat /etc/passwd (liste des comptes utilisateurs système) et uname -a (version du noyau Linux, utile pour identifier des CVEs d'escalade de privilèges kernel).

Étape 5 – Escalade de privilèges verticale via vim

J'ai exécuté sudo -l pour lister toutes les commandes qu'un utilisateur peut exécuter avec sudo, avec ou sans mot de passe. La sortie indiquait que l'utilisateur k1tt3ns pouvait exécuter /usr/bin/vim sans fournir de mot de passe (entry NOPASSWD: /usr/bin/vim). vim est un éditeur de texte interactif qui dispose d'une fonctionnalité permettant d'exécuter des commandes shell depuis son interface. En lançant vim avec sudo puis en utilisant la commande interne :!bin/sh, on obtient un shell hérité du processus vim, qui tourne avec les droits root : 

sudo /usr/bin/vim
# Dans vim, taper en mode commande :
:!bin/sh
# Résultat : shell root ouvert dans le terminal
whoami
# root

Cette technique est répertoriée dans GTFOBins, une base de données open source listant les binaires Unix pouvant être utilisés pour contourner les restrictions de sécurité locales. Elle s'applique à de nombreux outils interactifs : less (!sh), more (!sh), nano (^R^X), awk (BEGIN {system("/bin/sh")}), etc.

Étape 6 – Récupération du flag

Depuis le shell root, je me suis déplacé dans le répertoire home de l'utilisateur avec cd /home/k1tt3ns et j'ai listé les fichiers avec ls -la. J'ai découvert le fichier flag.sh. En l'exécutant avec ./flag.sh, le script m'a demandé d'entrer l'adresse IP de ma machine (10.15.250.129) et a affiché le flag validant la réussite complète de la session de pentesting.

AC24.01Cyber | Connaître et utiliser les bonnes pratiques et les recommandations de cybersécurité

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai réalisé une analyse complète du guide ANSSI Recommandations pour la protection des systèmes d'information essentiels (v1.0), qui définit 62 recommandations organisées en 8 thèmes. Chaque mesure a été confrontée à nos configurations réelles dans Packet Tracer et classée selon trois statuts : Retenue (mesure appliquée totalement ou partiellement), Exclue (pertinente mais techniquement impossible dans Packet Tracer, comme le MFA) et Non applicable (hors périmètre, comme les supports amovibles ou l'externalisation).

1. Bilan de conformité : 85 % des mesures retenues

Sur 62 recommandations, 53 ont été retenues (85 %), dont 35 totalement mises en œuvre et 18 partiellement. 5 mesures ont été exclues (R17+, R19, R33, R39, R50), uniquement en raison des contraintes de Packet Tracer. 21 mesures sont non applicables à ce type de projet. Ce taux de 85 % s'explique par la richesse de l'infrastructure déployée : double couche de filtrage (ACL switch L3 + ASA), tunnel GRE/IPSec chiffré, DMZ, HTTPS/TLS, DNSSEC, SSH avec Telnet désactivé, politique deny-all sur toutes les ACLs.

2. Points forts de l'implémentation

Segmentation et cloisonnement (R9, R10, R11) : le réseau est découpé en 3 VLANs par site (Admin VLAN10/11, Service VLAN20/21, Production VLAN30/31) plus une DMZ isolée (192.168.40.0/24 et 192.168.41.0/24). Chaque VLAN n'accède qu'aux ressources nécessaires à son activité. L'admin a un accès total, Service et Production sont totalement isolés.

Chiffrement inter-sites (R11-, R18, R22) : tunnel GRE over IPSec avec AES-256, SHA-256, IKEv1/ISAKMP et Diffie-Hellman Group 5. Les flux inter-sites sont chiffrés de bout en bout sur le réseau opérateur.

Sécurisation web et DNS (R16) : HTTPS/TLS avec certificat auto-signé RSA 4096 bits, redirection permanente HTTP → HTTPS, en-têtes HSTS (max-age=31536000), X-Frame-Options, X-XSS-Protection, X-Content-Type-Options, Referrer-Policy. DNSSEC avec ZSK RSA 2048 bits et KSK RSA 4096 bits, enregistrements RRSIG vérifiés.

Filtrage exhaustif (R23, R24, R25, R27, R28) : politique deny-all sur toutes les ACLs (switch L3 et ASA), matrice de flux documentée en section 10 du Write Up, règles commentées et justifiées, hitcounts vérifiables avec show access-list.

3. Mesures exclues et plans d'action production

Les 5 mesures exclues disposent toutes d'un plan d'action documenté pour un déploiement en production réelle. R17+ et R19 (MFA accès public et nomade) : déployer TOTP (Google Authenticator / FreeOTP) ou certificat client TLS ; R33 (MFA admin) : Cisco ISE ou FreeRADIUS + OTP pour l'authentification SSH ; R39 (réseau physique d'administration dédié) : câblage physique séparé remplaçant le VLAN Admin logique ; R50 (coffre-fort de mots de passe) : KeePassXC (usage individuel) ou HashiCorp Vault (usage centralisé).

AC24.02Cyber | Mettre en œuvre les outils fondamentaux de sécurisation d'une infrastructure du réseau

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai déployé et configuré une double couche de sécurisation sur les deux sites de l'entreprise : des ACLs stateful sur les switchs L3 et des pare-feux Cisco ASA 5506-X avec politique d'inspection.

1. ACL BLOCK_SERVICE_PROD sur switch L3 — whitelist stateful

La première couche de filtrage est une ACL étendue nommée BLOCK_SERVICE_PROD, appliquée en direction in sur les interfaces VLAN20 (Service) et VLAN30 (Production) des switchs SR1 et SR2. Elle adopte une approche whitelist stricte : seules les initiations de l'Admin et les réponses vers l'Admin sont autorisées, tout le reste est bloqué. 

ip access-list extended BLOCK_SERVICE_PROD
 permit ospf any any                              ! OSPF non bloqué
 permit ip   192.168.10.0 0.0.0.255 any          ! Admin gauche : accès total
 permit ip   192.168.11.0 0.0.0.255 any          ! Admin droit  : accès total
 permit tcp  any 192.168.10.0 0.0.0.255 established ! Réponses TCP vers Admin gauche
 permit tcp  any 192.168.11.0 0.0.0.255 established ! Réponses TCP vers Admin droit
 permit icmp any 192.168.10.0 0.0.0.255 echo-reply  ! Réponses ping vers Admin gauche
 permit icmp any 192.168.11.0 0.0.0.255 echo-reply  ! Réponses ping vers Admin droit
 deny   ip   any any                              ! Tout le reste bloqué

Le mot-clé established filtre les paquets TCP dont le bit ACK ou RST est positionné, caractéristique des paquets appartenant à une connexion déjà établie. Cela autorise les réponses à des connexions initiées par Admin sans autoriser Service à initier lui-même de nouvelles connexions TCP. echo-reply fait de même pour ICMP.

2. Cisco ASA 5506-X — architecture trois zones

La seconde couche est un pare-feu Cisco ASA 5506-X par site, avec trois interfaces dédiées à trois zones de sécurité distinctes : 

interface GigabitEthernet1/1
 nameif outside        ! Zone WAN (vers routeur R1/R3)
 security-level 100
 ip address 172.16.1.1 255.255.255.0

interface GigabitEthernet1/2
 nameif inside         ! Zone LAN (vers switch L3)
 security-level 100
 ip address 172.16.0.2 255.255.255.0

interface GigabitEthernet1/3
 nameif DMZ            ! Zone DMZ (vers serveurs publics)
 security-level 100
 ip address 192.168.40.1 255.255.255.0

Trois ACLs nommées contrôlent chaque direction : INSIDE_IN (filtre le trafic LAN → ASA), OUTSIDE_IN (filtre le trafic WAN/tunnel → ASA), DMZ_IN (filtre le trafic DMZ → ASA). La règle fondamentale : permit ip 192.168.10.0 255.255.255.0 any donne un accès total à l'Admin, et deny ip any any en fin d'ACL bloque tout flux non explicitement autorisé.

3. inspect icmp — filtrage stateful ICMP

La commande inspect icmp dans la service-policy global_policy active le suivi d'état des sessions ICMP sur l'ASA. Quand un Admin envoie un ping, l'ASA crée une entrée dans sa table de sessions. La réponse echo-reply est automatiquement autorisée car elle correspond à la session. Si Service tente d'initier un ping, l'ASA ne trouve aucune session correspondante et le bloque, même sans règle ACL explicite. 

class-map inspection_default
 match default-inspection-traffic
policy-map global_policy
 class inspection_default
  inspect dns preset_dns_map
  inspect ftp
  inspect icmp
  inspect tftp
service-policy global_policy global
AC24.03Cyber | Sécuriser les services

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai sécurisé les services exposés en DMZ sur chacun des deux sites, en prenant en charge ma spécialisation individuelle : la migration HTTP vers HTTPS sur serveur Apache2 sous VM Linux Ubuntu.

1. Architecture DMZ — isolation des services publics

Chaque site dispose d'un serveur DMZ (DMZ SERVER 1 : 192.168.40.2 / DMZ SERVER 2 : 192.168.41.2) hébergeant trois services : HTTP (port 80), HTTPS (port 443) et DNS (port 53). Ces serveurs sont connectés sur l'interface Gig1/3 du pare-feu ASA (zone DMZ, 192.168.40.1 et 192.168.41.1), physiquement distincte du LAN interne (inside) et du WAN (outside). Si un serveur DMZ est compromis, l'attaquant se retrouve dans la DMZ, isolé du LAN interne par le pare-feu. L'ACL DMZ_IN contrôle ce que le serveur DMZ peut initier : il ne peut communiquer qu'avec les VLANs Admin des deux sites, jamais avec Service ou Production.

2. Sécurisation HTTPS avec Apache2 et OpenSSL (ma spécialisation)

Sur VM Linux Ubuntu (IP : 192.31.25.16), j'ai sécurisé le serveur web Apache2 en forçant TLS/SSL et en ajoutant les en-têtes de sécurité HTTP recommandés. 

# Installation et activation du module SSL
sudo apt install openssl
sudo a2enmod ssl

# Génération du certificat auto-signé RSA 4096 bits
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
    -days 365 -nodes \
    -subj "/C=FR/ST=Haut de france/L=Bethune/OU=R&T/CN=192.31.25.16"

# Copie dans le répertoire Apache
sudo cp cert.pem /etc/apache2/ssl/cert.pem
sudo cp key.pem  /etc/apache2/ssl/key.pem

Configuration dans /etc/apache2/sites-available/default-ssl.conf : 

ServerName 192.31.25.16
SSLEngine on
SSLCertificateFile    /etc/apache2/ssl/cert.pem
SSLCertificateKeyFile /etc/apache2/ssl/key.pem

# En-têtes de sécurité HTTP
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
Header set X-Content-Type-Options nosniff
Header set X-XSS-Protection "1; mode=block"
Header set X-Frame-Options SAMEORIGIN
Header set Referrer-Policy "strict-origin-when-cross-origin"

# Redirection permanente HTTP -> HTTPS
Redirect permanent / https://192.31.25.16/

Strict-Transport-Security (HSTS) force le navigateur à n'utiliser que HTTPS pendant un an, rendant impossible une attaque SSL Stripping. X-Content-Type-Options nosniff empêche le MIME sniffing (un fichier texte ne peut pas être exécuté comme script). X-Frame-Options SAMEORIGIN bloque le Clickjacking (chargement de la page dans une iframe d'un site tiers). X-XSS-Protection active le filtre XSS intégré aux navigateurs.

3. Sécurisation DNS avec DNSSEC (spécialisation de Matisse Rudiger)

DNSSEC (Domain Name System Security Extensions) assure l'intégrité et l'authenticité des enregistrements DNS via des signatures cryptographiques, empêchant les attaques de type cache poisoning. Deux clés sont générées : une ZSK (Zone Signing Key, RSA 2048 bits) qui signe les enregistrements de la zone, et une KSK (Key Signing Key, RSA 4096 bits) qui signe la ZSK, formant une chaîne de confiance. 

# Génération ZSK (RSA 2048 bits)
sudo dnssec-keygen -a RSASHA256 -b 2048 -n ZONE example.local
# Génération KSK (RSA 4096 bits)
sudo dnssec-keygen -f KSK -a RSASHA256 -b 4096 -n ZONE example.local

# Inclusion dans le fichier de zone et signature
sudo dnssec-signzone -o example.local /etc/bind/db.example.local

# Vérification — présence du flag 'ad' et des enregistrements RRSIG
dig @192.31.25.18 www.example.local +dnssec
# --> Réponse avec RRSIG + flag ad = validation DNSSEC réussie
AC24.04Cyber | Choisir les outils cryptographiques adaptés au besoin fonctionnel du système d'information

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai appliqué trois niveaux cryptographiques distincts, chacun adapté à un besoin fonctionnel précis : chiffrement des flux réseau inter-sites, sécurisation des services web en DMZ, et intégrité des enregistrements DNS.

1. Les trois grandes catégories cryptographiques

L'encodage (Base64, hexadécimal) transforme des données sans clé secrète. Il sert à la compatibilité de transport, jamais à la confidentialité. N'importe qui peut décoder un Base64 instantanément avec base64 -d.

Le hachage cryptographique (SHA-256, bcrypt, argon2id) est une fonction à sens unique irréversible. SHA-256 sert à la vérification d'intégrité (HMAC dans IPSec) ; bcrypt et argon2id servent au stockage de mots de passe car leur facteur de travail réglable rend le brute force impraticable.

Le chiffrement symétrique (AES-256) utilise une clé secrète partagée. C'est le plus performant pour de gros volumes de données : IPSec l'utilise pour chiffrer les flux réseau après échange de clés asymétrique. Le chiffrement asymétrique (RSA-4096, ECDSA) utilise une paire clé publique / clé privée : utilisé pour TLS/HTTPS, DNSSEC et l'échange de clés de session IPSec (Diffie-Hellman).

2. Tunnel GRE over IPSec — AES-256, SHA-256, IKEv1, DH Group 5

Le tunnel inter-sites entre R1 et R3 utilise IPSec pour chiffrer et authentifier tous les flux transitant sur le réseau opérateur. La combinaison des algorithmes garantit trois propriétés fondamentales : confidentialité (AES-256 en mode CBC chiffre les données, rendant le trafic illisible sur le réseau de transit R2), intégrité (HMAC-SHA-256 détecte toute altération des paquets) et authenticité (IKEv1/ISAKMP avec PSK et DH Group 5 assure que les deux extrémités sont bien R1 et R3). Le choix d'AES-256 plutôt qu'AES-128 est justifié par le contexte de sécurisation d'un SI essentiel (recommandation ANSSI R11-).

3. HTTPS avec certificat RSA 4096 bits — services DMZ

Pour les serveurs web en DMZ (ma spécialisation), j'ai choisi RSA 4096 bits plutôt que RSA 2048 bits car c'est la recommandation ANSSI pour les nouveaux certificats. La longueur de clé plus grande augmente la résistance aux attaques par factorisation. L'en-tête HSTS (Strict-Transport-Security: max-age=31536000) empêche toute dégradation vers HTTP non chiffré (attaque SSL Stripping), complémentant la redirection Apache. TLS assure ici le chiffrement asymétrique pour l'échange de clés (handshake), puis symétrique (AES) pour le flux de données.

4. DNSSEC — signatures RSA pour l'intégrité DNS

DNSSEC utilise le chiffrement asymétrique différemment : non pas pour la confidentialité, mais pour l'authenticité et l'intégrité des enregistrements DNS. La ZSK (RSA 2048 bits) signe les enregistrements de la zone avec sa clé privée ; les résolveurs vérifient avec la clé publique (DNSKEY). La KSK (RSA 4096 bits) signe la ZSK, formant une chaîne de confiance. Ce mécanisme prévient le DNS cache poisoning : un attaquant ne peut pas injecter de faux enregistrements car il ne possède pas la clé privée ZSK pour générer une signature RRSIG valide.

AC24.05Cyber | Connaître les différents types d'attaque

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai étudié et mis en pratique plusieurs types d'attaques réseau dans le cadre des tests de sécurité réalisés par Antoine Grevin, en participant à leur compréhension et à la définition des contre-mesures.

1. Attaque STP — Root Bridge Hijacking (Man-in-the-Middle L2)

Le Spanning Tree Protocol (STP, IEEE 802.1D) évite les boucles réseau en élisant un Root Bridge parmi tous les switchs : celui avec la priorité la plus basse est élu. SR2 était Root Bridge légitime avec priorité 32778. L'attaque consiste à connecter un switch malveillant (SW-A) sur un port trunk de SR2, puis à configurer sa priorité STP à 0 (valeur minimale) sur tous les VLANs : 

! Sur SW-A — switch attaquant
SW-A(config)# spanning-tree vlan 1  priority 0
SW-A(config)# spanning-tree vlan 10 priority 0
SW-A(config)# spanning-tree vlan 20 priority 0
SW-A(config)# spanning-tree vlan 30 priority 0

! Vérification — SW-A élu Root Bridge
SW-A# show spanning-tree vlan 10
Root ID  Priority 10        ← SW-A est Root Bridge
         This bridge is the root

! Sur SR2 — Root Bridge légitime déclassé
SR2# show spanning-tree vlan 10
Root ID  Priority 10
         Port 6(GigabitEthernet1/0/6)  ← Root via SW-A !

Conséquences : tout le trafic inter-VLANs transite désormais par SW-A, qui peut l'intercepter (Man-in-the-Middle L2). La reconvergence STP lors de l'élection provoque un déni de service de plusieurs secondes sur tous les VLANs. Un attaquant physiquement présent dans les locaux peut ainsi compromettre l'ensemble des communications sans toucher aux équipements d'administration.

2. Contre-mesures : BPDU Guard, PortFast et priorité fixe

! BPDU Guard sur tous les ports d'accès (non trunk)
SR2(config)# interface range GigabitEthernet1/0/2-5
SR2(config-if)# spanning-tree portfast
SR2(config-if)# spanning-tree bpduguard enable
! → Port mis en err-disabled si un BPDU est reçu

! Priorité STP fixe et élevée pour SR2 (garantit qu'il reste Root Bridge légitime)
SR2(config)# spanning-tree vlan 1,10,20,30 priority 4096

spanning-tree portfast active immédiatement les ports d'accès (postes clients) sans passer par les états Listening et Learning, éliminant le délai de 30 secondes au démarrage. spanning-tree bpduguard enable place le port en état err-disabled dès réception d'un BPDU : aucun switch non autorisé ne peut envoyer de BPDUs sur un port d'accès sans être immédiatement bloqué. La priorité 4096 sur SR2 garantit qu'aucun switch avec la priorité par défaut (32768) ne peut le détrôner comme Root Bridge.

3. Autres types d'attaques réseau étudiés dans ce contexte

VLAN hopping : un attaquant dans un VLAN tente d'atteindre un autre VLAN en exploitant le protocole 802.1Q. Contre-mesure : désactiver le mode trunk automatique (switchport nonegotiate), forcer le mode access sur tous les ports d'accès et définir un VLAN natif inutilisé.

Attaque par déni de service sur OSPF : injection de faux LSAs pour perturber les tables de routage. Contre-mesure dans notre infrastructure : OSPF est exempté des ACLs (permit ospf any any) mais les interfaces OSPF sont limitées aux équipements de confiance par le filtrage des réseaux déclarés dans chaque processus OSPF.

Interception de flux non chiffrés : sans le tunnel IPSec, tout le trafic inter-sites sur le réseau de transit R2 serait lisible en clair. C'est précisément pour contrer ce risque que GRE over IPSec (AES-256, SHA-256) a été choisi pour l'interconnexion inter-sites.

import socket
import threading
import json

SERVER_HOST = "127.0.0.1"
SERVER_PORT = 52300
MAX_LISTEN  = 5
NET_BUFFER  = 16384      # 16 Ko

def run_server():
    server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    # SO_REUSEADDR : évite l'erreur "Address already in use" au redémarrage
    server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server_socket.bind((SERVER_HOST, SERVER_PORT))
    server_socket.listen(MAX_LISTEN)
    print(f"[SERVER] Listening on {SERVER_HOST}:{SERVER_PORT}")

    # Thread de monitoring clavier (q pour quitter)
    monitor = threading.Thread(target=input_monitor, daemon=True)
    monitor.start()

    while True:
        conn, addr = server_socket.accept()
        print(f"[+] New connection from {addr}")
        ClientWorker(conn, addr).start()

AF_INET désigne la famille d'adresses IPv4. SOCK_STREAM spécifie TCP (orienté connexion, fiable, ordonné). SO_REUSEADDR est une option socket au niveau SOL_SOCKET : sans elle, si le serveur redémarre rapidement après une fermeture, le port reste en état TIME_WAIT pendant 2×MSL (Maximum Segment Lifetime, soit ~60 secondes) et le bind échoue. listen(MAX_LISTEN) définit la taille de la file d'attente des connexions TCP en cours d'établissement.

2. Thread ClientWorker (traitement concurrent)

class ClientWorker(threading.Thread):
    def __init__(self, conn, addr):
        super().__init__(daemon=True)
        self.conn = conn
        self.addr = addr

    def run(self):
        try:
            # Signaler au client que la connexion est établie
            self.conn.sendall("READY".encode(ENCODING))

            while True:
                raw = self.conn.recv(NET_BUFFER)
                if not raw:
                    break  # Déconnexion brutale du client
                message = raw.decode(ENCODING).strip()
                if message == DISCONNECT:
                    break

                # Désérialisation de la requête JSON
                request = json.loads(message)
                results = execute_search(
                    request.get("query1", ""),
                    request.get("query2", ""),
                    request.get("mode", "simple"),
                    request.get("extensions", [".txt",".pdf",".xlsx",".html"])
                )
                # Sérialisation et envoi de la réponse
                response = json.dumps(results, ensure_ascii=False)
                self.conn.sendall(response.encode(ENCODING))

        except (ConnectionResetError, BrokenPipeError) as e:
            print(f"[-] Client {self.addr} disconnected: {e}")
        finally:
            self.conn.close()
            print(f"[-] Connection with {self.addr} closed.")

daemon=True est essentiel : un thread daemon se termine automatiquement lorsque le programme principal se termine, évitant que des threads clients orphelins empêchent la fermeture propre du serveur. sendall() est préféré à send() car il garantit l'envoi complet des données même si le noyau fragmente l'envoi en plusieurs segments TCP. ConnectionResetError et BrokenPipeError couvrent les cas de déconnexion brutale (coupure réseau, kill du client).

3. Réception en boucle côté client (assemblage des chunks)

def receive_full_response(sock):
    """Receive all chunks until end of transmission."""
    full_data = b""
    while True:
        chunk = sock.recv(NET_BUFFER)
        if not chunk:
            break  # Socket fermé
        full_data += chunk
        if len(chunk) < NET_BUFFER:
            break  # Dernier chunk (taille < buffer = fin de transmission)
    return full_data.decode(ENCODING)

TCP est un protocole de flux : il ne préserve pas les frontières de messages. Un seul appel à send() côté serveur peut nécessiter plusieurs appels à recv() côté client si les données dépassent la taille du buffer réseau ou si la pile TCP décide de segmenter. La détection de fin de message utilise le fait qu'un chunk de taille inférieure au buffer (len(chunk) < NET_BUFFER) signale généralement la fin des données disponibles. Pour des protocoles plus robustes, on utiliserait des délimiteurs (\n) ou un en-tête de longueur.

1. Accès aux fichiers TXT avec cache LRU

from functools import lru_cache

@lru_cache(maxsize=128)
def get_file_lines(filepath):
    """Read a text file and return its lines as a tuple (hashable for cache)."""
    try:
        with open(filepath, "r", encoding="utf-8", errors="ignore") as f:
            return tuple(f.readlines())
    except (OSError, IOError) as e:
        print(f"[ERROR] Cannot read {filepath}: {e}")
        return tuple()

def search_txt(filepath, query1, query2, mode):
    lines = get_file_lines(filepath)
    results = []
    for line_num, line in enumerate(lines, start=1):
        if match_logic(line, query1, query2, mode):
            results.append({
                "file": filepath,
                "location": f"Line {line_num}",
                "context": line.strip()[:120]
            })
    return results

@lru_cache(maxsize=128) est un décorateur de la bibliothèque standard Python qui mémoïse les résultats de la fonction. Le cache stocke les résultats des 128 derniers appels distincts (combinaisons uniques d'arguments). Lorsque la même combinaison d'arguments est redemandée, la valeur est retournée directement depuis le cache sans ré-exécuter la fonction. Le fichier n'est lu depuis le disque qu'une seule fois pour toutes les requêtes portant sur ce fichier dans la même session serveur. La fonction retourne un tuple (immuable) plutôt qu'une liste car lru_cache exige que les valeurs de retour soient hashables.

2. Accès aux fichiers HTML avec BeautifulSoup

from bs4 import BeautifulSoup

def search_html(filepath, query1, query2, mode):

4. Logique de correspondance unifiée (match_logic)

import re

def match_logic(text, query1, query2, mode):
    """
    Unified matching logic for all file formats.
    Returns True if text matches the query according to mode.
    """
    source = str(text).lower()
    q1 = query1.lower().strip() if query1 else ""
    q2 = query2.lower().strip() if query2 else ""

    if mode == "simple":
        return bool(q1) and (q1 in source)

    elif mode == "and":
        return bool(q1) and bool(q2) and (q1 in source) and (q2 in source)

    elif mode == "or":
        if not q1 and not q2:
            return False
        return (bool(q1) and q1 in source) or (bool(q2) and q2 in source)

    elif mode == "regex":
        if not query1:
            return False
        try:
            return bool(re.search(query1, str(text), re.IGNORECASE))
        except re.error as e:
            print(f"[REGEX ERROR] Invalid pattern '{query1}': {e}")
            return False  # Pattern invalide : pas de crash serveur

    return False

Le bloc try/except re.error sur le mode regex est critique pour la robustesse du serveur : si un client envoie un pattern regex invalide (par exemple [ sans fermeture, ou * en début de pattern), re.search() lèverait une exception re.error qui, sans ce bloc, crasherait le thread ClientWorker et interromprait la connexion. Avec le bloc, le serveur retourne simplement aucun résultat sans crash. str(text) en entrée est nécessaire pour gérer les cellules Excel numériques (int, float, datetime) qui ne sont pas des chaînes de caractères nativement.

Dans le cadre de la SAÉ 3.01 – Mettre en œuvre un système de transmission, j'ai étudié, déployé et caractérisé plusieurs systèmes de transmission radio complexes, en combinant recherche bibliographique, manipulations pratiques sur des équipements SDR réels, et analyse spectrale.

Ses caractéristiques clés : plage de fréquences de 325 MHz à 3,8 GHz, largeur de bande instantanée maximale de 20 MHz, taux d'échantillonnage configurable jusqu'à 61,44 MSPS (méga échantillons par seconde), résolution ADC/DAC de 12 bits. 2. Prise en main de GNU Radio Companion

GNU Radio Companion (GRC) est un environnement de développement visuel par blocs pour les applications SDR. Un flowgraph GNU Radio est constitué de : sources (génèrent des échantillons : Signal Source, File Source, SDR Source), blocs de traitement du signal (Rational Resampler, Low Pass Filter, WBFM Receive, AGC, Multiply Const), et puits (consomment les échantillons : Audio Sink, File Sink, QT GUI Sink). Les blocs sont connectés par des liens représentant des flots d'échantillons complexes (I/Q – In-phase et Quadrature).

J'ai d'abord réalisé des exercices d'initiation (émetteur AM, récepteur AM) pour comprendre la logique de traitement du signal numérique, puis développé un récepteur FM complet.

3. Récepteur FM avec Adalm-Pluto et relation Fs

La difficulté principale de la réception FM avec un SDR est d'assurer la compatibilité entre le taux d'échantillonnage du SDR et le taux attendu par la carte son. La relation à respecter est : 

Fs = samp_rate / dRS × dAD

Où samp_rate est le taux d'échantillonnage de l'Adalm-Pluto (par exemple 2 048 000 Hz = 2,048 MSPS pour couvrir la bande FM), dRS est le facteur de décimation du bloc Rational Resampler (qui réduit le débit en gardant 1 échantillon sur dRS), et dAD est le facteur d'interpolation du second Resampler vers la carte son. Fs doit être égal à 48 000 Hz (48 kHz) ou 44 100 Hz (44,1 kHz) selon la carte son utilisée. Avec samp_rate = 2 048 000, dRS = 32, dAD = 3/4 : 

Fs = 2 048 000 / 32 × (3/4) = 64 000 × 0.75 = 48 000 Hz ✓

Le bloc WBFM Receive (Wide-Band FM Receiver) démodule le signal FM (déviation de fréquence ±75 kHz pour la FM broadcast). La chaîne complète : PlutoSDR Source → Low Pass Filter (coupe les canaux FM adjacents) → Rational Resampler (décimation) → WBFM Receive → Rational Resampler (interpolation vers 48 kHz) → Audio Sink.

4. Standard DAB+ et ses caractéristiques

Le DAB+ (Digital Audio Broadcasting Plus) utilise la modulation COFDM (Coded OFDM) avec 1536 sous-porteuses sur une largeur de bande de 1,536 MHz, espacées de 1 kHz. Le préfixe cyclique (guard interval) de 246 μs protège contre les évanouissements multichemins (réflexions sur les bâtiments). Le codec audio HE-AAC v2 offre une qualité équivalente à la FM pour 32 kbit/s seulement. Par rapport à la FM analogique (200 kHz de bande passante par station), DAB+ est 6 fois plus efficace spectralement. En France, DAB+ est déployé en bande III (174-240 MHz).

AC22.02 | Mettre en place un accès distant sécurisé

Dans le cadre de la SAÉ 3.Cyber.04 – Découvrir le pentesting, j'ai mis en place un accès distant sécurisé par SSH sur une machine cible en résolvant une problématique technique complexe liée au mécanisme de port knocking et au timing de l'établissement de connexion TCP.

1. Protocole SSH et mécanismes de sécurité

SSH (Secure Shell, RFC 4251-4254) est le protocole standard de connexion distante sécurisée. Il fonctionne sur le port TCP 22 par défaut et établit un canal chiffré en plusieurs phases : négociation de l'algorithme de clé (Diffie-Hellman, ECDH), échange des clés publiques de serveur, authentification du client (par mot de passe ou clé publique), puis ouverture du canal de données. Toutes les communications après l'échange initial sont chiffrées avec AES (symétrique) et authentifiées avec HMAC-SHA2. La première connexion à un serveur demande à l'utilisateur de vérifier et accepter la clé publique du serveur (fingerprint), qui est ensuite stockée dans ~/.ssh/known_hosts pour les connexions suivantes.

2. Port knocking : mécanisme et implémentation iptables

Le port knocking est implémenté via le module xt_recent d'iptables, qui maintient une liste d'adresses IP ayant récemment contacté des ports spécifiques. Un exemple de configuration iptables implémentant un port knocking en trois étapes (ports 7000, 8000, 9000) serait : 

# Étape 1 : garder trace des IPs ayant touché le port 7000
iptables -A INPUT -p tcp --dport 7000 -m recent --name KNOCK1 --set -j DROP
# Étape 2 : si l'IP a fait KNOCK1 ET touche le port 8000
iptables -A INPUT -p tcp --dport 8000 -m recent --name KNOCK1 --rcheck \
    -m recent --name KNOCK2 --set -j DROP
# Étape 3 : si l'IP a fait KNOCK2 ET touche le port 9000 → ouvrir SSH
iptables -A INPUT -p tcp --dport 9000 -m recent --name KNOCK2 --rcheck \
    -m recent --name OPEN --set -j DROP
# Autoriser SSH seulement si l'IP est dans la liste OPEN
iptables -A INPUT -p tcp --dport 22 -m recent --name OPEN --rcheck \
    --seconds 30 -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j DROP

L'option --seconds 30 ferme automatiquement l'accès SSH après 30 secondes si aucune connexion n'est établie. C'est précisément ce délai court qui créait la difficulté de synchronisation.

3. Solution technique : deux terminaux en parallèle

Le problème était que la fenêtre d'ouverture du port SSH durait seulement quelques secondes après le port knocking. En exécutant séquentiellement (port knocking, puis SSH), le port se refermait avant que la poignée de main TCP soit complète. La solution technique utilise deux terminaux simultanés :

AC22.03 | Mettre en place une connexion multi-site via un réseau opérateur

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai mis en place une interconnexion sécurisée entre deux sites distants en déployant un tunnel GRE over IPSec à travers un routeur de transit R2, simulant un réseau opérateur.

1. Architecture : tunnel GRE à travers un routeur de transit

Trois routeurs constituent l'infrastructure WAN : R1 (site gauche, 10.0.0.1), R2 (transit opérateur, 10.0.0.2 / 10.0.1.1) et R3 (site droit, 10.0.1.2). Le tunnel GRE s'établit directement entre R1 et R3 à travers R2. R2 ne connaît que les adresses de transit et reste ignorant de la topologie interne des deux sites, ce qui représente exactement le modèle d'un réseau opérateur réel : l'entreprise utilise l'infrastructure de transport sans lui exposer ses routes internes.

2. Configuration du tunnel GRE sur R1 et R3

! Configuration R1 (site gauche)
interface GigabitEthernet0/0
 ip address 10.0.0.1 255.255.255.0   ! Lien vers R2 (réseau opérateur)
 no shutdown
interface Tunnel0
 ip address 172.30.255.1 255.255.255.252
 tunnel source GigabitEthernet0/0    ! Source : 10.0.0.1
 tunnel destination 10.0.1.2         ! Destination : interface de R3 côté opérateur

! Configuration R3 (site droit)
interface GigabitEthernet0/0
 ip address 10.0.1.2 255.255.255.0   ! Lien vers R2 (réseau opérateur)
 no shutdown
interface Tunnel0
 ip address 172.30.255.2 255.255.255.252
 tunnel source GigabitEthernet0/0    ! Source : 10.0.1.2
 tunnel destination 10.0.0.1         ! Destination : interface de R1 côté opérateur

! Vérification du tunnel
R1# show interfaces Tunnel0
Tunnel0 is up, line protocol is up
R1# ping 172.30.255.2
!!!!!

3. Dual OSPF process — séparation réseau physique / tunnel

R1 et R3 utilisent deux processus OSPF distincts pour séparer proprement le routage du réseau de transit et le routage à travers le tunnel : 

! R1 — OSPF process 1 : réseau physique de transit
router ospf 1
 network 10.0.0.0   0.0.0.255 area 0  ! Lien R1 ↔ R2
 network 172.16.1.0 0.0.0.255 area 0  ! Lien R1 ↔ FIREWALL-1
 network 172.16.0.0 0.0.0.255 area 0  ! Inside FIREWALL-1 ↔ SR1

! R1 — OSPF process 2 : tunnel GRE inter-sites avec redistribution
router ospf 2
 redistribute ospf 1     ! Injecter les routes du réseau physique
 redistribute connected  ! Injecter les réseaux directement connectés
 network 172.30.255.0 0.0.0.3 area 0  ! Interface Tunnel0

La redistribution mutuelle entre OSPF 1 et OSPF 2 est le mécanisme clé : OSPF 2 apprend les routes internes (VLANs Admin/Service/Production, DMZ) via OSPF 1 et les propage à l'autre site à travers le tunnel. Sans cette redistribution, les réseaux internes resteraient invisibles de l'autre côté. Les pare-feux ASA participent également à OSPF 1 pour annoncer les réseaux LAN inside et DMZ.

4. Services rendus accessibles inter-sites grâce au tunnel

Grâce au tunnel GRE et à la redistribution OSPF, les VLANs Admin de chaque site peuvent accéder aux serveurs DMZ de l'autre site. Les enregistrements DNS croisés permettent à admin (site gauche) de résoudre dmz2.entreprise.local → 192.168.41.2 et d'accéder en HTTPS au serveur DMZ du site droit, et réciproquement. Cet accès inter-sites est validé par les tests nslookup et https://dmz2.entreprise.local depuis les postes Admin.

AC22.04 | Déployer des réseaux d'accès des opérateurs

Dans le cadre de la SAÉ 3.01 – Mettre en œuvre un système de transmission, j'ai étudié et analysé concrètement les réseaux d'accès déployés par les opérateurs télécoms, en observant leurs signaux directement avec des équipements de mesure.

1. Analyse du réseau mobile 4G/LTE

LTE (Long Term Evolution, 3GPP Release 8) est la technologie cellulaire dite "4G". Elle utilise OFDMA (Orthogonal Frequency Division Multiple Access) en voie descendante et SC-FDMA en voie montante. L'accès multiple est obtenu en allouant des Resource Blocks (RBs) à chaque terminal : un RB est constitué de 12 sous-porteuses OFDM × 7 symboles OFDM dans le domaine temps-fréquence, représentant 180 kHz × 0,5 ms. La bande passante d'un canal LTE peut être 1,4 / 3 / 5 / 10 / 15 / 20 MHz.

En utilisant MCS Spectrum Analyzer couplé au Spectran V4, j'ai capturé et analysé les signaux échangés entre mon terminal mobile et un relais 4G LTE proche de l'IUT de Béthune. J'ai pu identifier dans les captures : le canal de synchronisation (PSS/SSS – Primary/Secondary Synchronization Signal) permettant au terminal de détecter la cellule et se synchroniser dessus ; le canal de diffusion (PBCH – Physical Broadcast Channel) qui transmet les paramètres système (MIB : Master Information Block) ; les niveaux de puissance RSSI (Received Signal Strength Indicator) et RSRP (Reference Signal Received Power) indiquant la qualité de la réception.

2. Analyse Wi-Fi (IEEE 802.11)

J'ai analysé les réseaux Wi-Fi environnants à l'IUT avec MCS Spectrum Analyzer. En bande 2,4 GHz (IEEE 802.11b/g/n), j'ai observé les 14 canaux nominaux de 20 MHz espacés de 5 MHz, dont les canaux se chevauchent partiellement. Seuls les canaux 1, 6 et 11 sont non-chevauchants. Dans l'environnement IUT, plusieurs réseaux utilisaient des canaux adjacents (par exemple canaux 2 et 5), créant des interférences co-canal dégradant les performances. En bande 5 GHz (IEEE 802.11a/ac/ax), les canaux de 20 MHz sont espacés de 20 MHz (donc non chevauchants), offrant jusqu'à 23 canaux disponibles en Europe selon les bandes UNII.

Cette analyse illustre les défis auxquels font face les opérateurs de réseaux Wi-Fi (hotspots publics, Wi-Fi entreprise) : la bande 2,4 GHz est surchargée (aussi utilisée par Bluetooth, micro-ondes, ZigBee) tandis que la 5 GHz offre plus de bande passante mais avec une portée réduite (absorption plus forte par les matériaux).

3. Standard DAB+ (réseaux d'accès radio numériques)

Le DAB+ (Eureka 147, ETSI EN 300 401) est le standard de radiodiffusion numérique terrestre déployé en Europe par les opérateurs de radio. Architecture du signal DAB+ : chaque multiplex (ensemble) DAB occupe une largeur de bande de 1,536 MHz et peut transporter plusieurs programmes audio (10 à 20 selon les débits) grâce au multiplexage OFDM. Les sous-porteuses sont espacées de 1 kHz avec un préfixe cyclique de 246 μs protégeant contre les réflexions multichemins jusqu'à 246 km/s de différence de trajet. En France, les opérateurs DAB+ déploient sur la bande III (174-240 MHz) dans les grandes agglomérations. Le codec HE-AAC v2 offre une qualité FM pour 32 kbit/s (contre 192 kbit/s pour le MP3 de qualité équivalente), permettant de diffuser davantage de programmes par multiplex.

AC22.05 | Capacité à questionner un cahier des charges RT

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai analysé et questionné le cahier des charges pour concevoir une infrastructure réseau d'entreprise deux sites avec pare-feux, DMZ et VPN.

1. Identification et traduction des besoins en contraintes techniques

Le CDC imposait plusieurs exigences qui ont nécessité des choix architecturaux justifiés. Exigence de segmentation : l'entreprise comprend trois profils distincts (Admin, Service, Production) avec des niveaux de confiance différents → traduction : 3 VLANs par site avec ACLs différenciées et politique d'isolation stricte pour Service et Production. Exigence de services publics : des services web et DNS doivent être accessibles depuis les deux sites sans exposer le LAN interne → traduction : DMZ isolée sur une troisième interface ASA, serveurs DMZ séparés physiquement du LAN. Exigence d'interconnexion sécurisée : les deux sites doivent communiquer sans exposer la topologie interne à l'opérateur → traduction : tunnel GRE over IPSec avec OSPF dual-process.

2. Choix architecturaux justifiés

Pourquoi une DMZ et non exposer les serveurs directement dans le LAN ? Si un serveur web en LAN est compromis, l'attaquant accède directement à tous les VLANs internes. Avec une DMZ, il est cloisonné dans le réseau 192.168.40.0/24, isolé du LAN par le pare-feu ASA. L'ACL DMZ_IN limite ce que le serveur peut initier depuis la DMZ, limitant les mouvements latéraux.

Pourquoi une double couche de filtrage (ACL switch L3 + ASA) et non uniquement le pare-feu ? Une seule couche ASA ne filtrerait pas les flux locaux entre VLANs avant qu'ils n'atteignent le pare-feu. La couche ACL sur le switch L3 bloque les initiations dès la sortie du VLAN source (au plus proche de la source, meilleure pratique Cisco), réduisant la charge du pare-feu et ajoutant une défense en profondeur : la compromission d'une couche ne suffit pas à atteindre les ressources.

Pourquoi GRE over IPSec et non simplement un peering BGP avec l'opérateur ? BGP exposerait les routes internes sur un réseau partagé. GRE encapsule et IPSec chiffre : R2 (opérateur) ne voit que des paquets IPSEC opaques entre 10.0.0.1 et 10.0.1.2 et n'a aucune connaissance des réseaux 172.16.x.x, 192.168.x.x et des DMZ.

AC21.01 | Configurer et dépanner le routage dynamique dans un réseau

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai configuré le routage dynamique OSPF sur l'ensemble de l'infrastructure réseau deux sites avec une architecture à deux processus distincts sur les routeurs de bordure.

1. OSPF sur les switchs L3 (SR1 et SR2)

OSPF (Open Shortest Path First) est un protocole de routage à état de liens qui construit une carte complète de la topologie réseau et calcule les meilleurs chemins avec l'algorithme de Dijkstra. Sur les switchs L3, le routage IP est activé (ip routing) et les sous-réseaux de chaque VLAN sont déclarés : 

! SR1 — site gauche
ip routing
router ospf 1
 network 172.16.0.0   0.0.0.255 area 0  ! Lien vers FIREWALL-1
 network 192.168.10.0 0.0.0.255 area 0  ! VLAN Admin
 network 192.168.20.0 0.0.0.255 area 0  ! VLAN Service
 network 192.168.30.0 0.0.0.255 area 0  ! VLAN Production

! SR2 — site droit (symétrique)
router ospf 1
 network 172.16.10.0  0.0.0.255 area 0
 network 192.168.11.0 0.0.0.255 area 0
 network 192.168.21.0 0.0.0.255 area 0
 network 192.168.31.0 0.0.0.255 area 0

2. Dual OSPF process sur R1 et R3 — séparation transit / tunnel

! R1 — OSPF 1 : réseau physique de transit
router ospf 1
 network 10.0.0.0   0.0.0.255 area 0  ! Lien R1 ↔ R2
 network 172.16.1.0 0.0.0.255 area 0  ! Lien R1 ↔ FIREWALL-1

! R1 — OSPF 2 : tunnel GRE avec redistribution
router ospf 2
 redistribute ospf 1
 redistribute connected
 network 172.30.255.0 0.0.0.3 area 0  ! Interface Tunnel0

! FIREWALL-1 — OSPF 1 : annonce ses réseaux LAN et DMZ
router ospf 1
 network 172.16.0.0   255.255.255.0 area 0  ! Inside ↔ SR1
 network 172.16.1.0   255.255.255.0 area 0  ! Outside ↔ R1
 network 192.168.40.0 255.255.255.0 area 0  ! DMZ

Le dual OSPF process évite d'exposer les réseaux internes (VLANs, DMZ) dans le processus de transit. OSPF 1 gère le réseau partagé avec l'opérateur ; OSPF 2 gère le tunnel GRE et utilise la redistribution pour propager les routes internes entre les deux sites.

4. HSRP et relais DHCP — SAÉ 3.Cyber.03

Dans la SAÉ 3.Cyber.03 – Concevoir un réseau informatique sécurisé multi-sites, j'ai approfondi la configuration du routage dynamique avec HSRP pour la redondance des passerelles et le relais DHCP pour la centralisation de l'attribution d'adresses.

HSRP (Hot Standby Router Protocol) crée une passerelle virtuelle partagée entre deux switchs de distribution (SR1_siege et SR2_siege). SR1 est configuré avec une priorité de 105 et la preemption activée, ce qui le désigne comme routeur actif préféré et lui permet de reprendre ce rôle automatiquement après une panne : 

! SR1_siege — routeur actif préféré (priorité 105)
interface Vlan10
 mac-address 0001.976d.b901     ! MAC fixe pour stabiliser les tables ARP
 ip address 192.168.10.2 255.255.255.0
 ip helper-address 192.168.10.10 ! Relais DHCP vers Windows Server
 standby 10 ip 192.168.10.1     ! IP virtuelle = passerelle des clients
 standby 10 priority 105        ! > 100 (défaut SR2) → SR1 actif
 standby 10 preempt             ! Reprend le rôle actif au retour

! SR2_siege — routeur standby (priorité par défaut 100)
interface Vlan10
 mac-address 0002.976d.b901
 ip address 192.168.10.3 255.255.255.0
 ip helper-address 192.168.10.10
 standby 10 ip 192.168.10.1
 standby 10 priority 100

La directive ip helper-address 192.168.10.10 sur chaque interface VLAN convertit les broadcasts DHCP des clients (UDP port 67) en unicast vers le Windows Server (192.168.10.10), permettant à un seul serveur DHCP centralisé de servir tous les VLANs. Le champ giaddr de la requête DHCP relayée contient l'adresse de l'interface SVI entrante, ce qui permet au serveur DHCP d'identifier le VLAN d'origine et de répondre avec une adresse du scope correspondant.

AC21.02 | Configurer et expliquer une politique simple de QoS et la sécurité d'un réseau

Dans le cadre de la SAÉ 4.Cyber.01 – Sécuriser un système d'information, j'ai conçu et configuré une politique de sécurité réseau en deux couches : ACLs stateful sur les switchs L3 et ACLs sur les pare-feux Cisco ASA avec filtrage stateful ICMP.

1. Couche 1 — ACL BLOCK_SERVICE_PROD sur switch L3 (whitelist stateful)

Cette ACL, appliquée en direction in sur les interfaces VLAN20 et VLAN30 des deux switchs SR1 et SR2, adopte une politique whitelist stricte. Les mots-clés established et echo-reply permettent uniquement les réponses à des connexions initiées par Admin, jamais les initiations depuis Service ou Production : 

ip access-list extended BLOCK_SERVICE_PROD
 permit ospf any any                               ! OSPF non bloqué
 permit ip   192.168.10.0 0.0.0.255 any           ! Admin gauche : accès total
 permit ip   192.168.11.0 0.0.0.255 any           ! Admin droit  : accès total
 permit tcp  any 192.168.10.0 0.0.0.255 established ! Réponses TCP → Admin gauche
 permit tcp  any 192.168.11.0 0.0.0.255 established ! Réponses TCP → Admin droit
 permit icmp any 192.168.10.0 0.0.0.255 echo-reply  ! Réponses ping → Admin gauche
 permit icmp any 192.168.11.0 0.0.0.255 echo-reply  ! Réponses ping → Admin droit
 deny   ip   any any                               ! Tout le reste bloqué

! Application sur les VLANs Service et Production
interface Vlan20
 ip access-group BLOCK_SERVICE_PROD in
interface Vlan30
 ip access-group BLOCK_SERVICE_PROD in

2. Couche 2 — ACLs ASA et inspect icmp stateful

Le pare-feu ASA gère trois ACLs nommées. La règle fondamentale : Admin a un permit ip any, tout le reste se termine par deny ip any any. L'instruction inspect icmp maintient un état des sessions ICMP : une réponse ping est autorisée uniquement si la requête initiale provenait d'Admin. Un ping initié par Service est bloqué car l'ASA ne trouve aucune session correspondante. 

! FIREWALL-1 — INSIDE_IN (trafic LAN → ASA)
access-list INSIDE_IN extended permit ip 192.168.10.0 255.255.255.0 any ! Admin : tout
access-list INSIDE_IN extended deny   ip any any

! FIREWALL-1 — OUTSIDE_IN (trafic WAN/tunnel → ASA)
access-list OUTSIDE_IN extended permit tcp any host 192.168.40.2 eq www   ! HTTP DMZ
access-list OUTSIDE_IN extended permit tcp any host 192.168.40.2 eq 443   ! HTTPS DMZ
access-list OUTSIDE_IN extended permit udp any host 192.168.40.2 eq 53    ! DNS DMZ
access-list OUTSIDE_IN extended permit ip 192.168.11.0 255.255.255.0 192.168.10.0 255.255.255.0
access-list OUTSIDE_IN extended permit icmp any 192.168.10.0 255.255.255.0 echo-reply
access-list OUTSIDE_IN extended deny   ip any any

! inspect icmp — filtrage stateful ICMP
class inspection_default
 inspect icmp
service-policy global_policy global
